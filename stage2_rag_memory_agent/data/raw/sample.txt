# RAG、Embedding 与 Memory 学习资料

RAG 是 Retrieval-Augmented Generation 的缩写，中文常译为“检索增强生成”。它的核心思想是：在大模型回答问题之前，先从外部资料中检索相关内容，再把检索到的内容作为上下文交给大模型，让模型基于资料生成答案。

普通大模型回答问题时，主要依赖模型参数中已有的知识。如果问题涉及模型没有学过的新资料，或者涉及本地文件、公司文档、个人笔记等私有资料，模型可能无法准确回答。RAG 的作用就是把外部资料接入大模型，让模型能够基于指定资料回答问题。

一个基础 RAG 流程通常包括五步：第一步是读取文档，第二步是将文档切分成多个 chunk，第三步是将 chunk 转换成 embedding 向量，第四步是根据用户问题检索最相关的 chunk，第五步是把检索结果和用户问题一起交给大模型生成答案。

chunk 是文档切分后的文本片段。之所以要切分文档，是因为大模型的上下文窗口有限，不能总是把整篇文档全部传给模型。如果文档很长，直接传入模型会浪费 token，也容易让模型抓不住重点。通过 chunk，可以只把和问题最相关的内容传入模型。

chunk 的大小会影响 RAG 效果。chunk 太短，可能缺少上下文；chunk 太长，可能包含太多无关信息。实际项目中，常见做法是让 chunk 之间保留一定 overlap，也就是重叠内容，这样可以减少重要信息被切断的问题。

Embedding 是一种把文本转换成向量的技术。向量可以理解为一组数字，用来表示文本的语义特征。语义相似的文本，在向量空间中的距离通常更近；语义差异较大的文本，在向量空间中的距离通常更远。

Embedding 检索的基本做法是：先把每个文档 chunk 转换成 embedding 向量并保存起来；当用户提出问题时，再把用户问题也转换成 embedding 向量；然后计算用户问题向量和每个 chunk 向量之间的相似度，选出最相关的 chunk。

retrieve 是检索步骤。它的输入通常是用户问题和文档索引，输出是若干个最相关的 chunk。关键词检索主要看字面是否匹配，而 embedding 检索更关注语义是否相近。因此，即使用户问题和原文用词不完全一样，embedding 检索也可能找到相关内容。

RAG 回答时不应该让模型自由发挥，而应该要求模型优先依据检索到的资料。如果资料中没有答案，模型应该明确回答“根据现有资料无法确定”，而不是编造答案。这是 RAG 系统降低幻觉的重要方式。

citations 是引用来源。一个严谨的 RAG 系统不仅要给出答案，还应该说明答案来自哪个文档、哪个 chunk 或哪个链接。引用来源可以帮助用户检查答案依据，也可以帮助开发者排查检索结果是否正确。

Memory 是 Agent 的记忆能力。短期记忆通常指当前对话上下文，长期记忆通常指保存到文件、数据库或向量库中的信息。长期记忆可以帮助 Agent 记住用户偏好、长期目标、历史背景和之前的重要事实。

Memory 和 RAG 的作用不同。RAG 主要用于从外部资料中查找事实依据，例如从文档、网页、知识库中找答案。Memory 主要用于保存和读取与用户相关的长期信息，例如用户希望用中文解释代码、用户正在学习 Python Agent 开发、用户偏好一步一步讲解。

一个完整的 Agent 可以同时使用 RAG 和 Memory。RAG 提供资料证据，Memory 提供用户背景。比如用户问“接下来我该学什么”，Agent 可以先读取长期记忆，知道用户正在学习 Python Agent 开发；再通过 RAG 检索学习资料；最后综合用户背景和资料内容给出建议。

RAG、Memory 和 Tool Use 是 Agent 开发中的三个重要能力。RAG 让 Agent 能查资料，Memory 让 Agent 能记住用户，Tool Use 让 Agent 能调用外部工具完成任务。把这三种能力组合起来，就可以构建更接近真实应用的 Agent。
